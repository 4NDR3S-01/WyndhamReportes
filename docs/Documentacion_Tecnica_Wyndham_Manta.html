1. Resumen del proyecto

Sistema web interno para el Hotel Wyndham Manta desarrollado con Laravel 13.8 + Filament 5.6. Consta de dos módulos principales:

MóduloPropósito
CocinaGestión de consumo de alimentos, importación de datos desde Excel, análisis de productos y generación de reportes.
MédicoGestión de atenciones médicas (partes diarios), pacientes, inventario de medicinas/insumos (Kardex), catálogos clínicos y reportes.

El sistema está completamente localizado al español. La UI es responsiva y soporta modo oscuro. Se accede mediante un panel de administración web con inicio de sesión seguro.

El proyecto sigue la estructura estándar de Laravel con las siguientes adiciones específicas:

Nota importante: Este proyecto no usa Filament Resources. Cada página es un componente Livewire full-page que extiende Filament\Pages\Page. El estado (formularios, filtros, paginación) se maneja con propiedades públicas de PHP. Las operaciones CRUD usan Model::query()->updateOrCreate() y validación inline, no Form Objects ni Resource tables.

Al guardar una atención, el sistema automáticamente:

  3. Descuenta los medicamentos del inventario creando movimientos de salida en el kardex.

Al eliminar una atención, el sistema revierte los movimientos de inventario asociados y elimina los registros de medicación, manteniendo la integridad del stock.

Filtros

8. Sistema de Kardex

8.1 Arquitectura del Kardex

El sistema de inventario (Kardex) sigue una estructura de tres niveles:

MedicoProducto │ ├── 1┼→ MedicoKardex (saldo mensual por producto) │ │ │ └── 1┼→ MedicoKardexMovimiento (mov. individual) │ ├── tipo: ingreso | salida | ajuste │ ├── origen: manual | parte_diario │ └── vinculado a MedicoParteDiario │ ├── medicamento_id → Medicamento (catálogo clínico) │ └── MedicoProductoAlias (nombres alternativos) MedicoKardex (atributos) ├── fecha_inicio / fecha_fin (período) ├── tipo: medicina | insumo ├── nombre ├── saldo_anterior, ingresos, egresos, total └── fecha_caducidad, hash_unico

PropiedadDescripción

Gestión completa de inventario médico:

9. Servicios

9.1 ProcesadorArchivoConsumo

Ubicación: app/Services/Cocina/ProcesadorArchivoConsumo.php

Servicio que procesa archivos Excel/CSV del módulo Cocina. Flujo interno:

  1. Lectura: Carga el archivo con PhpSpreadsheet y obtiene todas las filas.
  2. Detección de encabezados: Busca filas que contengan "Fecha" y "Cantidad" (normalizado).
  3. Normalización: Parsea fechas (múltiples formatos), números (conversión de comas/puntos) e infiere el servicio (desayuno/almuerzo/cena) desde el concepto.
  4. Resolución de productos: Busca por código o nombre, actualiza si existe o crea uno nuevo.
  5. Deduplicación: Usa hash_unico SHA-256 (archivo + fila) más verificación de duplicados exactos por fecha+producto+servicio+concepto.
  6. Registro de errores: Las filas inválidas se registran como errores por separado.

9.2 InventarioMedicoService

Ubicación: app/Services/Medico/InventarioMedicoService.php

Registro de movimientos de inventario médico. Características principales:

  • Asegura la existencia de un registro MedicoKardex para el producto antes de crear el movimiento.
  • Soporta tres orígenes: manual (desde inventario) y parte_diario (desde atención médica).
  • Vincula movimientos de salida con el parte diario que los originó para trazabilidad.

9.3 KardexMensualService

Ubicación: app/Services/Medico/KardexMensualService.php

Generación de reportes kardex bajo demanda (sin persistencia):

  • Calcula saldos desde movimientos reales en medico_kardex_movimientos.
  • Reemplazó las tablas redundantes medico_kardex_cierres / medico_kardex_cierre_items (eliminadas en migración).
  • Para cada producto calcula: saldo anterior (desde kardex previo + movimientos), ingresos, egresos (incluyendo ajustes positivos/negativos) y total.

12.2 Importación de archivos Cocina

  • Formatos: .xlsx, .xls, .csv
  • Tamaño máximo: 10 MB
  • Almacenamiento: storage/importaciones/cocina/
  • Detección dinámica: Los encabezados se detectan automáticamente por nombre normalizado (ej: "cantidad", "nombrearticulo", "presentacion").
  • Deduplicación: Doble verificación: hash SHA-256 por archivo+fila, más duplicado exacto por fecha+producto+servicio+concepto.

Archivo de datos semilla: assets/DEPARTAMENTO_MEDICO.xlsx — contiene múltiples hojas con catálogos, pacientes y datos iniciales.

15. Migración a producción

El script migrate_to_mysql.php copia datos desde la base SQLite local hacia una base MySQL vacía de producción.

Variables de entorno requeridas

SOURCE_SQLITE_PATH=            # Ruta al archivo SQLite de origen
TARGET_DB_HOST=                # Host de la base MySQL
TARGET_DB_PORT=3306            # Puerto MySQL
TARGET_DB_DATABASE=            # Nombre de la base de datos
TARGET_DB_USERNAME=            # Usuario MySQL
TARGET_DB_PASSWORD=            # Contraseña MySQL

Proceso paso a paso

  1. En el servidor de producción, ejecutar: php artisan migrate --force para crear el esquema vacío.
  2. En el entorno local (con SQLite), definir las variables de entorno de producción.
  3. Ejecutar: php migrate_to_mysql.php
  4. Verificar que los datos se copiaron correctamente.
Advertencia: La base de producción debe estar vacía (solo esquema) antes de ejecutar la migración. El script inserta datos, no actualiza registros existentes.
